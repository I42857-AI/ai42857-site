三、命名与受众
================================================================================

### 两层受众设计（沙箱版）

脚本体系分两层，每层面向不同受众，命名规则完全不同：

| 层级 | 文件类型 | 受众 | 命名规则 | 示例 |
|------|----------|------|----------|------|
| 代码层 | `.sh` / `.js` / `.mjs` / `.py` | Agent / 工具链 | 拼音字母化 + **应用名+动作** | `Skills-TongBu.sh` |
| 启动器层 | `.sh` | Agent / Bash | 拼音字母化 + **应用名+动作** | `QClaw-YiTi.sh` |

**⚠️ 应用名维度**：功能一致但应用不同的脚本，必须用应用名区分。
- ✅ `Skills-TongBu.sh` vs `QClaw-ChongDong.sh`（同为同步/虫洞化，不同应用）
- ❌ `TongBu.sh`（无应用名，不知道同步谁的）

**⚠️ 层级隔离铁律**：
- 代码层和启动器层的内容**保持拼音字母化**（与 Windows 版兼容）
- **禁止跨层引用**：启动器层不得包含复杂代码逻辑
- 沙箱简化：Linux UTF-8 默认，中文不会乱码，但保持字母化命名规范

**字母化规则**（沙箱简化版）：
- Linux UTF-8 默认，中文文件名不会乱码
- 但保持拼音字母化以兼容 Windows 版本和工具链
- .sh 文件名：拼音字母化 PascalCase + kebab
- .sh 文件内容：变量名/函数名用拼音字母化，注释可用中文

### 命名规则

- .sh 文件名：`{应用名}-{动作}` 语义化命名，中文拼音 PascalCase + kebab
- **禁止数字编号前缀**：统一用「应用名+动作」语义化命名
- 沙箱特有脚本：
  - `Skills-TongBu.sh` — 技能虫洞化同步
  - `ShaXiang-HuiFu.sh` — 沙箱一键恢复

================================================================================
编码安全核心规则（详解见 CanKao/03-bianma-anquan.md）
================================================================================

  中文编码（沙箱简化版）：
    - Linux 默认 UTF-8，无中文乱码风险
    - .sh 文件名+内容 → 拼音字母化（保持兼容）
    - 读写含中文的文本/JSON → Python3 或 Node.js 均可
    - Symlink → ln -s（Linux 原生支持）

  双引号变量展开铁律（Bash 版）：
    变量后紧跟冒号等非变量名字符 → 必须用 ${} 包裹
    ❌ `echo "$AppName: OK"` → 可能解析错误
    ✅ `echo "${AppName}: OK"`

  动态检测（沙箱环境，禁止硬编码）：
  | 变量 | 获取方式 |
  |------|----------|
  | 工作区根目录 | /workspace/（固定） |
  | 数据源 | /workspace/.agents/（固定） |
  | 用户名 | `whoami` 或 `$USER` |
  | Node.js | `which node` |
  | Python | `which python3` |
  | 北京时间 | `TZ=Asia/Shanghai date '+%Y-%m-%d %H:%M:%S'` |

  Symlink 唯一类型 = SymbolicLink（ln -s，禁止 Junction/硬链接）
